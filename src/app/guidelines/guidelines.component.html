The 3D-Beacons Network provides unified programmatic access to experimentally determined and predicted structure models.

Terms of collaboration

The terms below set out the guidelines for the collaboration between data providers of 3D-Beacons. The terms will be reviewed annually at the Annual General Meetings.

3D-Beacons Hub

  • PDBe-KB will maintain data exchange format schema(s)
  • The schema will evolve in consultation with partner data resources
  • PDBe-KB will support the infrastructure for the 3D-Beacons Hub and Registry
  • The 3D-Beacons Hub will provide programmatic access to models of partner data resources
  • The 3D-Beacons Hub will link back to the partner data resources, transparently attributing credit for their contributions
  • PDBe-KB will maintain an open-access library of reusable data visualisation components
  • PDBe-KB and the collaborating partners define a transparent ranking of the models based on agreed criteria

Collaborating partners

  • The data provided will be freely available for academic and commercial use under Creative Commons Attribution 4.0 (CC-BY 4.0) license terms.
  • Data providers are responsible for the quality of the data they contribute and should provide quality metrics as part of their data. For theoretical models, a confidence measure is required according to the types supported in the API specification (can be extended in accordance with the 3D-Beacons Consortium).
  • Data providers must provide an email address as the contact point. The 3D-Beacons Hub will send a notification to this email address in case of problems with data access.
  • Data providers must provide metainformation for their models, including but not limited to the date of creation and the version of the underlying protein sequence.
  • Data providers must provide example accessions (e.g. UniProt accessions) which will always point to existing models.
  • Data providers must ensure that their API endpoints, if serving pregenerated data, have response rates lower than 10 seconds.
  • Data providers are encouraged to indicate at their services that they are parts of the 3D-Beacons Network, either by displaying the logo or a URL https://3d-beacons.org
  • Active data providers are required to attend annual 3D-Beacons Network meetings.
  • The protocols for model or structure generation must have associated peer-reviewed publications.
  • In the case of databases, the database must have a corresponding peer-reviewed publication.
  • In the case of models generated as part of a study, it is strongly encouraged to have a peer-reviewed publication that describes both the methodology and the usage.
  • The 3D-Beacons Network will remove inaccessible 3D-Beacons after six months of inaccessibility.
  • Data providers can settle any potential disputes through direct communication with the leaders of 3D-Beacons or by bringing the dispute to the annual general meeting.
  • The data contributed to 3D-Beacons by collaborating partners will be free from any restrictions on distribution and re-use under the CC BY 4.0 license or CC BY SA 4.0.
  • The partners are responsible for the quality of the data they contribute
  • Protocols for data generation of theoretical models must be published in peer-reviewed publications
  • The partners commit to updating the API implementations within two months of announcing major version updates of the API specification.
    1. The API specification uses semantic versioning, e.g. v.1.3.9, where 1. is the major version, 3. is the minor version and 9. is a patch.
    2. The Hub API will aggregate the current (live) API endpoints.
    3. The development Hub API will test the development API endpoints.
  • The partners commit to keeping Beacon API response times under 2 seconds for pre-calculated data.
    1. The 3D-Beacons Hub API enforces a 5 seconds timeout, i.e. Beacons not returning data within 5 seconds will not have their data exposed.
    2. New partner resources will have their API stress-tested by PDBe-KB.

General Data Protection Regulation (GDPR) notice

The 3D-Beacons Network collects no personal data from the contributing data providers, except for nominated email addresses that will serve the sole purpose of contact points in case of technical issues (e.g. when a 3D-Beacon server is inaccessible). Contributing data providers are responsible for implementing and following their own GDPR policies.

The 3D-Beacons Hub API endpoints, such as shown above, can be used to programmatically access all the available protein structures and models from any members of the 3D-Beacons Network. For example, the API endpoint for the SARS-CoV-2 polyprotein, identified by its UniProt accession P0DTD1 provides all the experimentally determined structures from the PDB, as well as theoretical models from SWISS-MODEL and others.
